Electronic Document

Management System

The product is designed to automate document-oriented business processes (BP)

EDMS

All information in the system is considered in the context of documents, which in turn are divided into types, classes and subclasses. The type of document determines its nature in the system, for example, the document can be an electronic form of data input / output according to a template, a file in Word, PDF or any other, electronic card, spreadsheet.

The class and subclass of the document is tied to the subject area, for example, class - "Contracts with customers", subclass1 "Contract for services", subclass2 "Contract for the supply of goods", or class "Administrative documents", subclass "Order of appointment" . Documents are provided to the user in the "Explorer" interface in the form of registers, which are automatically cataloged by classes and subclasses of documents. In addition, the user has the ability to create their own document folders within a subclass.

In addition to the body of the document that contains the content, the document has a card and tasks related to the actions on this document. The document card contains standard information related to the document flow: author, document numbers and dates of registration, etc., list of signatories, mailing list, list of appendices to the document, history of document processing and others.

Documents can be combined into a package of documents and participate in the BP as a single object.

 

Classification and cataloging

of information by document classes

The sequence of signing, approval and approval of the document is hardcover. The process is accompanied by a corresponding notification to the actors and is displayed in the document card. The document is signed electronically by various providers.

Managing

the signature, approval and adoption of the document

A task is assigned to a system user to perform actions related to document processing (document section), creating new documents, or performing system functions within the PSU. The mechanism of tasks is the basis of BP operation in the system. The task has its own life cycle and, within it, mechanisms for creating, assigning, monitoring and confirming the task. Tasks are cascaded into subtasks and used to plan works, decompose them and assign them to executors.

The movement of tasks creates the movement of documents attached to the tasks. This creates a document flow within the BP.

 

Manage

document-related and / or document-creating tasks

This applies to documents such as "electronic form created by a template" (abbreviated - "template"). One or more document templates are created for each document subclass. The template contains information about the type and location of data input / output fields, as well as other elements of the document such as section headings, document card fields and others.

Data input fields are text fields of fixed size or text blocks of unlimited size. Data output fields - one and two-dimensional tables and graphs (charts), which are built on the table.

Templates are formed using the appropriate input form designer. Output (printed) forms of the document are formed on the input automatically.

The document created by the template is stored in the fields of the database tables. It is displayed in the register of documents in the corresponding class.

For template documents, collective editing is possible with the result displayed on-line.

Document rights (read / edit access) are granted to system users by the author of the document. In the case of a template document, the rights are granted to the document section.

The application covers the network of command posts. Due to the distributed architecture, parts of the parent System work autonomously at each control post and, if necessary, can be integrated into common network, using encrypted messaging (multi-server interaction). The architecture of the System has been designed in accordance with the recommendations of NATO ARCHITECTURE FRAMEWORK (NAF) Ver 3. According to NAF, a service model and service-oriented architecture (hereinafter SOA) were accepted as the basis for decomposition of the System. The structure and composition of the functions of the System were implemented in accordance with the recommendations of NATO C3 Taxonomy Baseline 2.0. Connection to the set of automation methods has been carried out according to STANAG 5525 (“Joint C3 Information Exchange Data Model - JC3IEDM”).
